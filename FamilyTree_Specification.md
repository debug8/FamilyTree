# Family Tree — Специфікація та план розробки

Настільний застосунок на **C# / WPF** для збереження, редагування та візуалізації фамільного (генеалогічного) дерева з автоматичним визначенням ступеня родинного зв'язку між будь-якими двома особами.

Документ структуровано так, щоб кожен розділ і кожну задачу можна було передавати LLM як самостійне технічне завдання. Задачі пронумеровані (T-x.y), мають чіткі вхідні умови, результат і критерії приймання.

---

## 1. Загальний опис і цілі

### 1.1. Призначення
Користувач веде базу осіб своєї родини: додає людей, вказує родинні зв'язки (батько/мати — дитина, подружжя), а застосунок:
- будує графічне дерево **вгору** (предки) або **вниз** (нащадки) від обраної особи;
- показує **всіх** родичів обраної особи — прямих і непрямих (бічні гілки: брати/сестри, дядьки, племінники, двоюрідні тощо);
- для кожної відображеної особи підписує **назву родинного зв'язку** відносно кореневої особи (наприклад: «тітка», «племінник», «двоюрідна тітка», «троюрідний брат», «прабаба»).

### 1.2. Ключові сценарії використання (Use Cases)
| ID | Сценарій |
|----|----------|
| UC-1 | Додати нову особу з персональними даними |
| UC-2 | Редагувати / видалити особу |
| UC-3 | Додати зв'язок «батько/мати → дитина» |
| UC-4 | Додати зв'язок «подружжя» |
| UC-5 | Обрати кореневу особу та побудувати дерево предків (вгору) |
| UC-6 | Побудувати дерево нащадків (вниз) |
| UC-7 | Побудувати повне дерево родичів (предки + нащадки + бічні гілки) |
| UC-8 | Побачити назву родинного зв'язку кожної особи відносно кореневої |
| UC-9 | Запитати ступінь родства між двома довільними особами |
| UC-10 | Пошук особи за прізвищем/ім'ям |
| UC-11 | Зберегти родину у файл `.familytree`; відкрити існуючий файл |
| UC-12 | Злиття з іншим файлом родини; опційно імпорт/експорт GEDCOM |
| UC-13 | Перемкнути мову інтерфейсу (українська / англійська) — включно з назвами родства |

### 1.3. Нефункціональні вимоги
- **Платформа:** Windows 10/11, .NET 10 (LTS), WPF. (.NET 10 — актуальний LTS, підтримка до листопада 2028; .NET 8/9 виходять з підтримки 10.11.2026, тому для нового проєкту обрано 10.)
- **Архітектура:** MVVM (рекомендовано `CommunityToolkit.Mvvm`).
- **Зберігання:** модель «документ»: одна родина = один файл `.familytree` (JSON усередині). Весь граф завантажується в пам'ять; збереження — атомарний перезапис файлу. Без СУБД (обґрунтування — розділ 3.5). Доступ до сховища — тільки через інтерфейс `IFamilyStorage`, щоб за потреби замінити реалізацію на SQLite без змін у домені та UI.
- **Продуктивність:** коректна робота з файлом до ~5 000 осіб; побудова дерева до 8–10 поколінь без помітних затримок (< 1 c); збереження файлу < 200 мс.
- **Локалізація:** застосунок мультимовний від початку. Мова за замовчуванням — українська, друга мова — англійська (як доказ працездатності механізму). Жодного рядка, видимого користувачу, в коді чи XAML напряму; назви родства — через `IKinshipFormatter` окремо для кожної мови; перемикання мови — з налаштувань без перезапуску (або з м'яким перезапуском вікна). Деталі — розділ 2.4.
- **Тестованість:** доменна логіка (модель, алгоритм родства) — в окремому проєкті без залежності від WPF; покриття unit-тестами (xUnit + Shouldly; моки — NSubstitute, лише для ViewModel-ів).

## 2. Архітектура рішення

### 2.1. Структура solution
```
FamilyTree.sln
├── FamilyTree.Domain        // Сутності, доменні сервіси, алгоритм родства. Без залежностей від UI.
├── FamilyTree.Storage       // IFamilyStorage + JsonFamilyStorage: серіалізація, атомарний запис, бекапи.
├── FamilyTree.App           // WPF: Views, ViewModels, конвертери, стилі, ресурси локалізації, DI-контейнер.
└── FamilyTree.Tests         // xUnit + Shouldly (+ NSubstitute для ViewModel-ів): домен, сховище, локалізація.
```

### 2.2. Принципи
- **MVVM:** View (XAML) ↔ ViewModel (команди, стан) ↔ Domain/Storage. Code-behind — тільки для суто візуальних речей (zoom/pan на Canvas).
- **DI:** `Microsoft.Extensions.DependencyInjection` + `Microsoft.Extensions.Hosting`; реєстрація `IFamilyStorage`, доменних сервісів, `ILocalizationService`, ViewModel-ів.
- **Документна модель:** відкритий файл живе в пам'яті як `FamilyDocument` (особи + зв'язки + метадані); ViewModel-и працюють з ним напряму, `IFamilyStorage` викликається лише на Open/Save. Прапорець «є незбережені зміни» (*) у заголовку вікна.
- **Асинхронність:** читання/запис файлу — `async`; UI не блокується.
- **Валідація:** на рівні домену (інваріанти) + на рівні ViewModel (`INotifyDataErrorInfo`).

### 2.3. Основні компоненти
| Компонент | Шар | Відповідальність |
|-----------|-----|------------------|
| `Person`, `ParentChildLink`, `SpouseLink` | Domain | Сутності |

### 2.4. Механізм локалізації
Три категорії текстів, у кожної — свій механізм:

1. **Статичні рядки UI** (написи, кнопки, заголовки, повідомлення валідації): `.resx`-ресурси в `FamilyTree.App/Resources` — `Strings.resx` (українська, базова) + `Strings.en.resx`. Доступ із XAML — через markup extension `{loc:Localize Key}` або статичний клас-обгортку; з коду — через `ILocalizationService.GetString(key)`. Ніяких literal-рядків у XAML/коді (перевіряється код-рев'ю; опційно — Roslyn-аналізатор).
2. **Назви родства** — це не переклад, а **окрема логіка для кожної мови** (англійська система інша: first cousin once removed ≠ буквальний переклад «двоюрідного дядька»). Тому: інтерфейс `IKinshipFormatter` у Domain; реалізації `UkrainianKinshipFormatter`, `EnglishKinshipFormatter`; вибір реалізації — за поточною культурою через DI-фабрику. Тести — окремо на кожен форматер.
3. **Формати дат і чисел** — через `CultureInfo` поточної мови (`DateOnly.ToString(culture)`), без ручного форматування.

Правила:
- Поточна мова зберігається в `settings.json`; при старті виставляється `Thread.CurrentThread.CurrentUICulture`.
- Перемикання мови в налаштуваннях: **зафіксовано (T-0.2) — живе оновлення без перезапуску**. Markup extension `{loc:Localize Key}` повертає прив'язку до індексатора класу-проксі `LocalizationSource` (реалізує `INotifyPropertyChanged`); `ILocalizationService.SetLanguage` піднімає подію `LanguageChanged`, за якою проксі сигналить оновлення індексатора (`Binding.IndexerName`) — і всі написи перечитуються миттєво. Дати/числа оновлюються через прив'язки до властивостей ViewModel, що реагують на `LanguageChanged`.
- Ключі ресурсів — ієрархічні: `Person_Editor_Title`, `Validation_CycleDetected`, `Tree_Mode_Ancestors`.
- Ідентифікація мови — за **власним кодом** застосунку (`LanguageOption.Code`), а не за `CultureInfo.TwoLetterISOLanguageName` (той не унікальний і для рідкісних мов повертає трилітерний код). Форматування дат/чисел — за окремим `FormattingCulture`, який має відкат, якщо реальної `CultureInfo` немає. Невідомий/битий код мови в `settings.json` не викидає виняток — тихий відкат на мову за замовчуванням.
- Додавання нової мови — два шляхи: (а) **вбудована** (є `CultureInfo`): новий `Strings.xx.resx` + `RegisterBuiltIn("xx")` + новий `XxKinshipFormatter`; (б) **без `CultureInfo`** (діалект/штучна/рідкісна мова): файл перекладу `%AppData%\FamilyTree\languages\<code>.json` (`displayName`, необов'язковий `formattingCulture`, `strings`), який підхоплюється при старті; відсутні ключі беруться з нейтральної мови. Деталі — `docs/localization-custom-languages.md`. Жодних змін у логіці.

## 3. Модель даних

### 3.1. Сутність `Person`
| Поле | Тип | Обов'язк. | Примітка |
|------|-----|-----------|----------|
| `Id` | Guid | так | PK |
| `LastName` | string(100) | так | Прізвище |
| `FirstName` | string(100) | так | Ім'я |
| `MiddleName` | string(100) | ні | По батькові |
| `MaidenName` | string(100) | ні | Дівоче прізвище |
| `Gender` | enum `Gender { Male, Female, Unknown }` | так | Потрібен для назв родства (дядько/тітка) |
| `BirthDate` | DateOnly? | ні | Може бути невідома |
| `BirthPlace` | string(200) | ні | |
| `DeathDate` | DateOnly? | ні | |
| `IsAlive` | bool (обчислюване) | — | `DeathDate == null` |
| `PhotoPath` | string? | ні | Копія фото в папці даних застосунку |
| `Notes` | string? | ні | Довільні нотатки |
| `CreatedAt` / `UpdatedAt` | DateTime | так | Аудит |

### 3.2. Зв'язки
**`ParentChildLink`** — спрямоване ребро «батько/мати → дитина»:
| Поле | Тип | Примітка |
|------|-----|----------|
| `Id` | Guid | PK |
| `ParentId` | Guid | FK → Person |
| `ChildId` | Guid | FK → Person |
| `ParentRole` | enum `ParentRole { Biological, Adoptive, Step }` | На старті достатньо `Biological`; enum закладаємо одразу |

**`SpouseLink`** — ненапрямлене ребро «подружжя»:
| Поле | Тип | Примітка |
|------|-----|----------|
| `Id` | Guid | PK |
| `Person1Id` / `Person2Id` | Guid | FK → Person; зберігати з `Person1Id < Person2Id` для унікальності |
| `MarriageDate` | DateOnly? | |
| `DivorceDate` | DateOnly? | `null` = шлюб чинний |

### 3.3. Інваріанти та валідація (доменні правила)
1. У особи не більше **одного біологічного батька** та **однієї біологічної матері** (перевірка за `Gender` батьків + `ParentRole = Biological`).
2. Заборона **циклів**: особа не може бути власним предком (перевірка DFS-ом по predkam при додаванні `ParentChildLink`).
3. Заборона дубля ребра (той самий `ParentId`+`ChildId`; та сама пара в `SpouseLink`).
4. Особа не може бути в шлюбі сама з собою; не може бути батьком самої себе.
5. М'які попередження (не блокують збереження, але показуються): дитина народжена до 12-річчя батька/матері; дата смерті раніша за дату народження; батько/мати молодші за дитину.

### 3.4. ER-діаграма (текстово)
```
Person 1 ──< ParentChildLink >── 1 Person   (Parent → Child)
Person 1 ──< SpouseLink >── 1 Person        (симетричний зв'язок)
```

### 3.5. Формат зберігання: файл `.familytree` (JSON)
СУБД не використовується. Обґрунтування: застосунок все одно тримає весь граф у пам'яті (алгоритм родства та розкладка дерева працюють з повним `FamilyGraph`), обсяг даних малий (5 000 осіб ≈ 2–3 МБ), а модель «родина = документ» природна для користувача: файл можна копіювати, синхронізувати через хмару, надсилати родичам.

**Структура файлу:**
```json
{
  "schemaVersion": 1,
  "meta": { "title": "Родина Шевченків", "createdAt": "...", "updatedAt": "...", "appVersion": "1.0.0" },
  "persons": [ { "id": "...", "lastName": "...", "firstName": "...", "gender": "Male", "birthDate": "1954-03-01", ... } ],
  "parentChildLinks": [ { "parentId": "...", "childId": "...", "parentRole": "Biological" } ],
  "spouseLinks": [ { "person1Id": "...", "person2Id": "...", "marriageDate": null, "divorceDate": null } ]
}
```

**Обов'язкові правила надійності:**
1. **Атомарний запис:** серіалізація в тимчасовий файл поруч → `File.Replace` (або rename) поверх цільового. Падіння процесу під час збереження ніколи не псує наявний файл.
2. **Версія схеми:** поле `schemaVersion`; читання файлу з новішою версією — зрозуміла відмова; зі старішою — ланцюжок міграторів `IFormatMigration (v1→v2→…)`.
3. **Резервні копії:** перед перезаписом — копія в підпапку `.backups` поруч із файлом (тримати останні 5, ротація).
4. **Фото** зберігаються не в JSON, а в папці `<ім'я файлу>.media/` поруч; у JSON — відносні шляхи.
5. Серіалізація — `System.Text.Json` (camelCase, enum-и рядками, `WriteIndented = true` для дружності до diff/git).

## 4. Алгоритм визначення ступеня родства (ядро застосунку)

### 4.1. Постановка
Для двох осіб **A** (коренева, «я») та **B** треба визначити: *ким є B для A* — і повернути українську назву з урахуванням статі B.

### 4.2. Крок 1: пошук найближчого спільного предка (НСП)
1. BFS-ом вгору по ребрах «дитина → батьки» зібрати для A множину предків з відстанями: `ancA[person] = a` (сама A має відстань 0).
2. Те саме для B: `ancB[person] = b`.
3. Спільні предки = перетин множин. Обрати НСП з мінімальною сумою `a + b` (при рівності — з мінімальним `max(a,b)`).
4. Якщо спільного предка немає — перевірити зв'язок через шлюб (див. 4.5), інакше «родинний зв'язок не встановлено».

> Особливий випадок: якщо B ∈ ancA, то НСП = B (b = 0); якщо A ∈ ancB — НСП = A (a = 0).

### 4.3. Крок 2: класифікація за парою (a, b)
`a` — відстань від A до НСП, `b` — відстань від B до НСП.

| Умова | Хто B для A | Назви (чол. / жін.) |
|-------|-------------|---------------------|
| a=0, b=1 | дитина | син / дочка |
| a=0, b=2 | онук | онук / онука |
| a=0, b=3 | правнук | правнук / правнучка |
| a=0, b≥4 | (b−2)-пра-онук | пра(пра…)внук — префікс «пра» × (b−2) |
| a=1, b=0 | батько/мати | батько / мати |
| a=2, b=0 | дід/баба | дід / баба |
| a=3, b=0 | прадід | прадід / прабаба |
| a≥4, b=0 | пра…дід | префікс «пра» × (a−2) |
| a=1, b=1 | сиблінг | брат / сестра (див. 4.4 — рідні/зведені) |
| a=2, b=1 | дядько/тітка | дядько / тітка |
| a=3, b=1 | двоюрідний дід | двоюрідний дід / двоюрідна баба |
| a=1, b=2 | племінник | племінник / племінниця |
| a=1, b=3 | внучатий племінник | внучатий племінник / внучата племінниця |
| a=2, b=2 | двоюрідний сиблінг | двоюрідний брат / двоюрідна сестра |
| a=3, b=2 | двоюрідний дядько | двоюрідний дядько / двоюрідна тітка |
| a=2, b=3 | двоюрідний племінник | двоюрідний племінник / двоюрідна племінниця |
| a=3, b=3 | троюрідний сиблінг | троюрідний брат / троюрідна сестра |

**Загальна формула** для a≥1, b≥1: нехай `k = min(a, b)`, `d = |a − b|`:
- Порядок «юрідності» = `k` → префікс: k=1 — «рідний» (не пишеться), k=2 — «двоюрідний», k=3 — «троюрідний», k=4 — «чотириюрідний», далі — «N-юрідний».
- Базове слово залежить від `d` та напрямку:
  - `d = 0` → брат / сестра;
  - `b < a` (B на старшій гілці): d=1 → дядько/тітка; d=2 → дід/баба; d≥3 → пра…дід (префікс «пра» × (d−2)) — з тим самим «юрідним» префіксом;
  - `b > a` (B на молодшій гілці): d=1 → племінник/племінниця; d=2 → внучатий племінник; d≥3 → правнучатий племінник (префікс «пра» × (d−2)).

Реалізувати як чисту функцію:
```csharp
KinshipResult Compute(Person a, Person b, FamilyGraph graph);
// KinshipResult: { RelationCode, DisplayNameUа, PathToCommonAncestor, StepsUp (a), StepsDown (b) }
```
Назву генерувати окремим форматером за інтерфейсом `IKinshipFormatter` (реалізації `UkrainianKinshipFormatter`, `EnglishKinshipFormatter` — див. 2.4 та 4.7), який приймає `(k, d, direction, gender)` і повертає рядок.

### 4.4. Рідні / зведені сиблінги
Для a=1, b=1: якщо в A і B збігаються **обидва** біологічні батьки — «рідний брат/сестра»; якщо тільки батько — «єдинокровний»; тільки мати — «єдиноутробний» (у UI допустимо спрощено: «зведений брат/сестра по батькові/матері»).

### 4.5. Свояцтво (зв'язки через шлюб) — етап 2 (опційно, окрема задача)
Якщо кровного зв'язку немає, шукати шлях виду `A —(кров)— X —(шлюб)— B` або `A —(шлюб)— X —(кров)— B`:
| Зв'язок | Назва |
|---------|-------|
| подружжя | чоловік / дружина |
| батько/мати дружини | тесть / теща |
| батько/мати чоловіка | свекор / свекруха |
| чоловік дочки / дружина сина | зять / невістка |
| брат чоловіка / сестра чоловіка | дівер / зовиця |
| брат дружини / сестра дружини | шурин / своячка |
| чоловік сестри | зять (шваґер) |
| дружина брата | невістка (братова) |
| подружжя дядька/тітки | «дружина дядька», «чоловік тітки» (описово) |
Якщо і так зв'язку немає — повертати «зв'язок не знайдено».

### 4.6. Крайні випадки
- A == B → «це та сама особа».
- Кілька НСП (подружжя-предки: спільні дід і баба) — результат однаковий, брати будь-якого з мінімальною сумою.
- Невідома стать → нейтральне формулювання: «дитина», «сиблінг», «родич 4-го коліна».
- Дуже далекий зв'язок (k > 5) → допустимо «далекий родич (N-юрідний)».

### 4.7. Англійський форматер (`EnglishKinshipFormatter`)
Алгоритм (пари (a, b)) — спільний для всіх мов; мовоспецифічна лише генерація назви. Англійська система:
- Пряма лінія: father/mother; grandfather/grandmother; префікс great- × (a−2): great-grandfather, great-great-grandfather. Аналогічно son/daughter, grandson, great-grandson.
- Сиблінги: brother/sister; half-brother/half-sister (без поділу на єдинокровних/єдиноутробних — в англійській його немає).
- d = 1, старша гілка: uncle/aunt; d = 2: granduncle (great-uncle)/grandaunt; d ≥ 3: great-granduncle…
- Молодша гілка: nephew/niece; grandnephew/grandniece; great-grandnephew…
- Кузени: формула cousin-ів: ступінь = k − 1 (first/second/third cousin), «removed» = d (once/twice removed). Приклад: (a=3, b=2) → first cousin once removed — те, що українською «двоюрідний дядько».
Ключовий висновок для архітектури: англійські назви **не мапляться 1:1** на українські (одному first cousin once removed відповідають і «двоюрідний дядько», і «двоюрідний племінник» залежно від напрямку) — саме тому форматер отримує сирі (k, d, напрямок, стать), а не перекладає готовий рядок.

## 5. Візуалізація дерева

### 5.1. Режими побудови (від кореневої особи R)
1. **Предки (вгору):** R → батьки → діди → … до вичерпання. Класичне «дерево предків» (кожен вузол має ≤ 2 батьків — виходить бінарне дерево догори).
2. **Нащадки (вниз):** R → діти → онуки → …; поруч із нащадком показувати його подружжя (другого з батьків його дітей).
3. **Повне дерево родичів:** обхід усього зв'язного компонента графа від R; кожна особа отримує підпис родства відносно R (через `KinshipCalculator`). Розкладка — за поколіннями: покоління особи = `StepsDown − StepsUp` відносно R (предки — вище, нащадки — нижче, сиблінги/кузени — на рівні R).

### 5.2. Розкладка (TreeLayoutEngine)
- Алгоритм: пошарова розкладка (layered) — спрощений Reingold–Tilford:
  1. кожній особі призначити рівень (покоління);
  2. рекурсивно порахувати ширину піддерев;
  3. батьків центрувати над дітьми (для режиму «вниз») / дітей під батьками;
  4. подружжя ставити поруч, з'єднувати горизонтальним ребром із «вузлом шлюбу», від якого йдуть лінії до дітей.
- Рушій повертає список `NodeLayout { PersonId, X, Y }` + список ребер `EdgeLayout { FromId, ToId, Kind }` — без WPF-типів.
- Рендеринг: `ItemsControl` на `Canvas` (вузли — DataTemplate, ребра — `Path`/`Line` в окремому шарі) або пряме малювання в `DrawingVisual` (швидше для великих дерев). Рекомендація: почати з `ItemsControl + Canvas`.

### 5.3. Вузол дерева (картка особи)
- Фото (або силует за статтю), «Прізвище Ім'я», роки життя «1954–2020» / «нар. 1988».
- **Підпис родства** відносно кореневої особи (кольоровий бейдж): «я», «мати», «двоюрідна тітка»…
- Клік — вибір; подвійний клік — зробити кореневою особою (перебудова дерева); контекстне меню: редагувати, додати дитину/батьків/подружжя, показати родство з…

### 5.4. Взаємодія з полотном
- Zoom колесом миші (масштаб 0.2–3.0, трансформація `ScaleTransform` відносно курсора).
- Pan перетягуванням правою кнопкою / серединою.
- Кнопки «Вписати в екран», «100%».
- Експорт видимого дерева в PNG (`RenderTargetBitmap`).

## 7. План робіт: етапи та задачі для LLM

Кожна задача сформульована так, щоб її можна було віддати LLM окремим промптом. До промпту завжди додавайте: (а) цей документ або відповідні розділи; (б) поточний код проєкту, який стосується задачі.

### Етап 0. Каркас проєкту
**T-0.1. Створити solution і проєкти.**
Створити структуру з розділу 2.1 (.NET 10). Підключити NuGet: `CommunityToolkit.Mvvm`, `Microsoft.Extensions.Hosting`, `xunit`, `Shouldly`, `NSubstitute`. Налаштувати DI в `App.xaml.cs` через Generic Host.
*Критерії:* solution компілюється; порожнє головне вікно запускається; тестовий проєкт бачить Domain і Storage.

**T-0.2. Інфраструктура локалізації + базові стилі.**
Реалізувати механізм із розділу 2.4: `Strings.resx` (uk, базова) + `Strings.en.resx`, `ILocalizationService` (поточна культура, `GetString`, подія зміни мови), markup extension `{loc:Localize Key}` для XAML, вибір мови в `settings.json`, перемикання через м'який перезапуск головного вікна. Плюс базові стилі кнопок/полів і палітра кольорів.
*Критерії:* жодного «зашитого» рядка в XAML головного вікна; перемикання uk ↔ en змінює всі написи; дата в статус-барі форматується за поточною культурою.

### Етап 1. Домен і збереження даних
**T-1.1. Доменні сутності.**
Реалізувати `Person`, `ParentChildLink`, `SpouseLink`, enum-и (розділи 3.1–3.2) у `FamilyTree.Domain`. Без зовнішніх залежностей.
*Критерії:* unit-тести на обчислюване `IsAlive`, рівність за Id.

**T-1.2. FamilyGraph.**
Клас, що будується зі списків осіб і зв'язків та дає O(1)-навігацію: `GetParents(id)`, `GetChildren(id)`, `GetSpouses(id)`, `GetSiblings(id)`, `GetConnectedComponent(id)`.
*Критерії:* тести на сімʼї з 3 поколінь, на зведених сиблінгів, на ізольовану особу.

**T-1.3. Валідація зв'язків.**
Сервіс `RelationshipValidator`: правила з розділу 3.3 (жорсткі помилки + м'які попередження), результат `ValidationResult { Errors[], Warnings[] }`. Тексти помилок — за ключами ресурсів (розділ 2.4), не literal-рядками.
*Критерії:* тести: цикл предків, третій біологічний батько, дубль ребра, дитина старша за батька (warning).

**T-1.4. Сховище: FamilyDocument + JsonFamilyStorage.**
Реалізувати за розділом 3.5: клас `FamilyDocument` (особи + зв'язки + метадані + `IsDirty`), інтерфейс `IFamilyStorage { LoadAsync(path), SaveAsync(doc, path) }`, реалізація `JsonFamilyStorage` на `System.Text.Json`: атомарний запис через тимчасовий файл + `File.Replace`, поле `schemaVersion` з ланцюжком міграторів, ротація 5 резервних копій у `.backups`, список останніх файлів у налаштуваннях.
*Критерії:* тести: save → load дає ідентичний документ; штучне переривання запису (виняток у середині серіалізації) не псує наявний файл; файл із `schemaVersion` більшою за підтримувану дає зрозумілу помилку; після 7 збережень у `.backups` рівно 5 копій.

### Етап 2. CRUD осіб і зв'язків (UI)
**T-2.1. Список осіб + пошук.**
`MainViewModel`: завантаження, живий пошук (debounce 300 мс), сортування. `ListBox`/`DataGrid` з шаблоном рядка (ім'я, роки).
*Критерії:* пошук «іва» знаходить «Іванов», «Іваненко»; список оновлюється після CRUD.

**T-2.2. Діалог створення/редагування особи.**
`PersonEditorViewModel` + діалог (розділ 6.2), валідація через `INotifyDataErrorInfo`, збереження фото.
*Критерії:* не можна зберегти без прізвища/імені/статі; фото копіюється і показується.

**T-2.3. Видалення особи.**
Підтвердження з переліком зв'язків, які буде видалено.
*Критерії:* після видалення граф і дерево коректно перебудовуються.

**T-2.4. Керування зв'язками.**
Діалог з розділу 6.3 + секція зв'язків у картці особи. Усі операції проходять через `RelationshipValidator`.
*Критерії:* сценарій «додати батьків, дітей, подружжя» повністю проходиться з UI; заборонені випадки дають зрозумілу помилку.

**T-2.5. Файлове меню документа.**
Меню Файл: Новий / Відкрити / Зберегти / Зберегти як / Останні файли; асоціація розширення `.familytree`; індикатор незбережених змін (*) у заголовку; запит на збереження при закритті; Ctrl+S.
*Критерії:* цикл «новий → додати осіб → зберегти → закрити → відкрити» відтворює дані повністю; закриття з незбереженими змінами завжди питає підтвердження.

### Етап 3. Алгоритм родства
**T-3.1. Пошук спільного предка.**
`CommonAncestorFinder` за розділом 4.2 (BFS, словники відстаней).
*Критерії:* тести: прямий предок, сиблінги, кузени, відсутність зв'язку, кілька НСП.

**T-3.2. KinshipCalculator + UkrainianKinshipFormatter.**
Класифікація (a, b) → код зв'язку → українська назва за таблицею і формулою розділу 4.3, стать з `Person.Gender`, рідні/зведені сиблінги (4.4).
*Критерії:* таблиця тестів ≥ 30 випадків, включно з: прабаба, внучата племінниця, двоюрідна тітка, троюрідний брат, чотириюрідний, «пра×3-дід».

**T-3.3. Пояснення шляху.**
Генерація людиночитного ланцюжка: «Оля — дочка Івана; Іван — брат Петра ⇒ Оля — племінниця Петра».
*Критерії:* для кузенів шлях проходить через НСП і читається природно.

**T-3.4. (Опційно) Свояцтво.**
Реалізувати розділ 4.5 як розширення калькулятора з окремим прапорцем «шукати зв'язки через шлюб».
*Критерії:* тесть/теща/свекор/свекруха/зять/невістка/дівер/шурин покриті тестами.

**T-3.5. EnglishKinshipFormatter.**
Друга реалізація `IKinshipFormatter` за розділом 4.7 (система cousin/removed, great-префікси); реєстрація фабрики форматерів за культурою в DI.
*Критерії:* тести ≥ 20 випадків: great-great-grandfather, first cousin once removed (в обох напрямках), second cousin, grandnephew, half-brother; перемикання мови в UI змінює бейджі родства без перезапуску логіки.

### Етап 4. Візуалізація
**T-4.1. TreeLayoutEngine.**
Розкладка за розділом 5.2 для трьох режимів; вхід — `FamilyGraph` + коренева особа + режим + глибина; вихід — координати вузлів і ребра. Юніт-тести на відсутність перекриттів вузлів одного рівня.
*Критерії:* для тестової сімʼї з 4 поколінь вузли не перетинаються; подружжя стоять поруч.

**T-4.2. TreeCanvasControl.**
Рендеринг вузлів (картка з 5.3) і ребер на Canvas; zoom/pan; вибір вузла; подвійний клік = нова коренева особа.
*Критерії:* плавний zoom до курсора; дерево на 200 осіб рендериться < 1 c.

**T-4.3. Панель керування деревом + підписи родства.**
`TreeViewModel`: режим, глибина, перебудова; для режиму «Усі родичі» кожен вузол отримує бейдж з назвою родства (виклик `KinshipCalculator` відносно кореня).
*Критерії:* перемикання Предки/Нащадки/Усі працює без перезапуску; бейджі відповідають тестовим випадкам T-3.2.

**T-4.4. Вкладка «Хто кому».**
Вибір двох осіб → назва зв'язку + пояснення шляху (T-3.3).
*Критерії:* для осіб без зв'язку — коректне повідомлення.

**T-4.5. Експорт дерева в PNG.**
`RenderTargetBitmap` всієї розкладки (не лише видимої частини).
*Критерії:* PNG відкривається, дерево повне й читабельне.

### Етап 5. Сервісні функції та поліровка
**T-5.1. Злиття родин (імпорт з іншого файлу).**
Оскільки формат зберігання і є JSON, окремий «експорт» не потрібен. Натомість: команда «Імпортувати з файлу…» — злиття осіб і зв'язків з іншого `.familytree` у відкритий документ із виявленням потенційних дублікатів (збіг ПІБ + дата народження) та звітом користувачу.
*Критерії:* імпорт файлу без перетинів додає всі записи; імпорт того самого файлу двічі не створює дублікатів після підтвердження злиття.

**T-5.2. (Опційно) Імпорт/експорт GEDCOM 5.5.1.**
Мінімальний профіль: INDI (NAME, SEX, BIRT, DEAT), FAM (HUSB, WIFE, CHIL, MARR).
*Критерії:* файл, експортований застосунком, відкривається в сторонній генеалогічній програмі.

**T-5.3. Обробка помилок і логування.**
Глобальний обробник виключень, Serilog у файл, дружні локалізовані повідомлення.

**T-5.4. Налаштування.**
Мова інтерфейсу, останні файли, тема, глибина дерева за замовчуванням — `settings.json` в AppData.
*Примітка:* мову й тему (світла/темна) реалізовано достроково в T-0.2 — живе перемикання через `ILocalizationService` та `IThemeService` (підміна тематичного `ResourceDictionary`; пензлі — через `DynamicResource`). Тут лишається додати решту полів.

**T-5.5. Тестові дані (seed).**
Команда «Створити демо-родину»: ~25 осіб, 4 покоління, кузени, зведені сиблінги, кілька шлюбів — для демонстрації всіх назв родства.
*Критерії:* після seed режим «Усі родичі» показує щонайменше 10 різних назв родства.

**T-5.6. Інсталятор / публікація.**
`dotnet publish -r win-x64 --self-contained` + (опційно) MSIX або Inno Setup.

## 10. Ризики та зауваги
- **Назви родства** — найчутливіша частина: обов'язково вести таблицю тестів (T-3.2, T-3.5) і поповнювати її при кожному знайденому неточному формулюванні. Пам'ятати, що назви для різних мов не мапляться 1:1 (розділ 4.7).
- **Розкладка дерева** для режиму «Усі родичі» складніша за класичні дерева (граф, а не дерево): на MVP допустимо групувати за поколіннями рядами й мінімізувати перетини ребер евристично.
- **Продуктивність Canvas:** якщо > 500 вузлів — перейти з ItemsControl на DrawingVisual (закладено в T-4.2 як план Б).
- **Цілісність файлу:** атомарний запис і бекапи (розділ 3.5) — не опція, а обов'язкова частина T-1.4; втрата родинного архіву — найгірший можливий баг застосунку.
- **Дані реальних людей:** дані зберігаються локально у файлі користувача; у документації зазначити, що вони нікуди не передаються.
